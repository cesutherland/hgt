## Task (verbatim from the issue body)

## Problem
`hgt work rm <n>` fails when run from *inside* an issue's worktree (e.g. its tmux
session):

    $ cd /home/carl/Projects/hgt-worktrees/61-executor-agent-denied-git-gh
    $ hgt work rm 61
    error: hgt work rm: no worktree for issue 61 under \
      /home/carl/Projects/hgt-worktrees/61-executor-agent-denied-git-gh-worktrees

The worktrees base is mis-derived — note the doubled `…-git-gh-worktrees` tail.

## Root cause
`_worktree_base()` (`cmds/work.sh:24-31`) computes the base from
`git rev-parse --show-toplevel`, which returns the **current worktree's** root, not the
main repo root. From inside worktree `…/61-<slug>`:
- `dirname`  → `…/hgt-worktrees`
- `basename` → `61-<slug>`
- result     → `…/hgt-worktrees/61-<slug>-worktrees`  ✗

So every worktree-relative path resolves wrong, and `_find_worktree` finds nothing.

## Fix
Resolve the **main worktree** root, not the current one. `git rev-parse
--path-format=absolute --git-common-dir` → `…/hgt/.git`; the main root is its `dirname`.
Base off that so it's correct from the main checkout *or* any worktree.

Sibling risk: `_repo_name()` (`cmds/work.sh:61`) uses the same `--show-toplevel` and is
wrong the same way from within a worktree (returns the worktree dir name, not `hgt`) —
masked in tests by `HGT_REPO_NAME`. Fix both off the common-dir root.
