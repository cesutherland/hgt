## Context
Local execution — today's `hgt work <n>`, and the planned local listener executor —
runs Claude Code with **full access to the machine**: `$HOME`, `~/.ssh`, the admin `gh`
auth, sibling repos, arbitrary FS. v0 reads the **raw, unnormalized issue body**, so a
prompt injection executes against the laptop. §3's "poor and powerless" was a property
of the *disposable Actions runner*; running locally throws it away. A sandbox is the
containment that makes local execution — attended or unattended — safe.

Carl wants this on the **critical path, solved first** (ahead of the listener), because
it also hardens the `hgt work` sessions he runs every day right now.

## Scope
Confine the Claude session `hgt work` spawns so it can reach **only**:
- the issue's worktree — not `$HOME`, not `~/.ssh`, not sibling repos, not arbitrary FS
- a **scoped** git/gh credential (machine-user PAT / scoped token) — never Carl's admin `gh`
- the network it needs (Anthropic API + the git remote), nothing wider

Pick a Linux mechanism (Carl's on Kubuntu). Candidates, decide by isolation-vs-friction:
- **bubblewrap (`bwrap`)** — namespace/FS jail wrapping the claude process; light, strong FS isolation.
- **container (podman/docker)** — only the worktree mounted; strongest, heavier, tmux ergonomics to solve.
- **dedicated low-priv unix user** — simple FS isolation, shares kernel/network.
- **Claude Code's own permission-mode/sandbox** — lightest, but tool-level not OS-level; likely insufficient alone.

Hard requirement: `tmux attach` into the sandboxed session must still work — the
"shell in if it gets dicey" ergonomics can't die.

## Parity
Identical sandbox on attended (`hgt work <n>`) and unattended (listener) paths — this is
#17 (attended/unattended parity) applied to the *execution environment*, not just the input.
