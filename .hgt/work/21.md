# Issue 21 — Phase 1: CI — run the conformance suite on PRs and gate merge on it

## Task (verbatim from the issue body)

**Phase 1 item** — dogfood-era hardening. Slot it into the local-loop phase, not Phase 0.

## Scope
Wire up GitHub Actions CI that runs the `bats` conformance suite on every PR to the default branch, and make it a **required status check** so a red suite blocks merge. Nothing else — this is the test-gate workflow, **not** the Phase 2 executor workflow.

## Why
Right now `test/run.sh` (bats) only runs when someone remembers to run it locally. The §3 merge discipline already requires a PR + human review; CI adds the other necessary-not-sufficient gate: **green tests**. Human stays the merge gate; CI just refuses to let red merge.

## Behavior
- **Trigger:** `pull_request` targeting the default branch (and optionally `push` to the default branch as a canary). Gate who can trigger per §3 action-hygiene — treat workflow files on PR branches as an escalation vector.
- **Job:** check out, install `bats`, run `test/run.sh` (which execs `bats test/`). The suite is hermetic (PATH-shims for `gh`/`git`/`tmux`/`claude` in `test/shims/`), so **no secrets, no network, no `ANTHROPIC_API_KEY`/`GITHUB_TOKEN` beyond default read** — keep this workflow poor and powerless (`permissions: contents: read`).
- **Gate:** register the job as a **required status check** in the default-branch ruleset so merge is blocked until it's green.

## Guardrail interplay (don't miss this)
The required check has to be added to the **existing §3 ruleset** (`forge_print_ruleset` in `lib/gh.sh`, the `pull_request` rule). Today that ruleset requires review but declares no `required_status_checks`. Either extend the printed ruleset to include the CI check, or document the manual step. Do **not** relax any other §3 setting to make CI fit.

## Open impl choices
- How to get `bats` in CI: `bats-core/bats-action`, `apt-get install bats`, or `npm i -g bats`. Lean on a pinned action; don't drift.
- Whether `hgt init`'s printed ruleset should learn about the CI check (couples init to a repo having CI) or stay decoupled — probably decoupled for now, note it.

## Non-goals
- The Phase 2 `claude-code-action` executor workflow (separate, has real write scope).
- Lint/format/typecheck jobs, matrix builds, caching — just run the suite.
- Auto-merge on green (§3: never; green is necessary, not sufficient).

## Exit criteria
- A PR to the default branch runs the suite and reports a status check.
- A failing suite **blocks** merge; passing + one human review allows it.
- The required-check wiring is reflected in the §3 ruleset (printed and/or applied), not bolted on out-of-band.
