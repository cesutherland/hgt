## Root cause (run 29969280003, working issue #62 — 15 denials, no PR)

Enumerated from the transcript's `permission_denials` array. **Zero of the 15 are on the
ship path** (git checkout/branch/add/commit/push, gh pr create) — the #61 allowlist was
never even exercised, because the agent never reached the push. The 15 split:

- **4 × Edit/Write to `.github/workflows/hgt-execute.yml`** (the actual blocker). The task
  *is* editing that file, and claude-code-action refuses writes under `.github/workflows/**`
  unless the token carries `workflows: write` — which we deliberately omit (§3). Two
  independent guardrails agree: even if the edit landed, `GITHUB_TOKEN` can't `git push`
  workflow changes without `workflows: write`. Granting it = the self-escalation §3 exists
  to prevent. So #62 is **structurally unshippable by the executor**, and correctly so.
- **~10 × research/probe flailing**: fetching the action's own `action.yml` (WebFetch not
  granted; `gh api`/`curl` not in allowlist; pipes/`;`/`&&` rejected as "multiple
  operations"; redirects outside cwd blocked) + `gh pr view`, `git log --all` chained with
  `rm`. None needed — the info was already in the workflow comments.

**Measurement error:** every cited run (29968194185: 9, 29969280003: 15) was the *same*
task #62. We've been diagnosing the allowlist against a task that can't ship regardless.
The "denials vary by task" framing is real but we never actually tested a shippable task.

## Fix

Minimal, per §3. **No allowlist widening** (no verb proven needed) and **no `workflows:
write`**. Instead:
- Prompt hardening: forbid fetching external URLs / the action's internals; bail loud when
  a required edit is refused (e.g. `.github/workflows/**`) instead of flailing + shipping a
  green no-op. Kills 13/15 of the observed denials.
- Document the root cause in the workflow so the next debugger doesn't re-chase the
  allowlist.
- Validation is a **throwaway `ready` issue that needs only normal code changes** — the
  real ship-path test #69 asks for. #62 (fail-loud guard) is the complementary fix and must
  land via a human/local session, since the executor cannot edit its own workflow.
