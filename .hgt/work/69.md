## Root cause — the allowlist dropped Edit/Write

The decisive evidence is run **29971041608 / issue #50** (a docs edit to `cmds/work.sh`, a
plain file): 14 denials, **every one an `Edit`/`Write` to `cmds/work.sh`** or a Bash
write-workaround (`sed -i`, `python`, `printf >`). The executor could branch/commit/push/
open a PR but **could not modify a single file**, so it "succeeded" green having shipped
nothing.

Why: `--allowedTools` **replaces** the default policy, it is not additive. #61 listed only
the git/gh ship verbs and thereby removed `Edit`/`Write` from the allowed set. Read-only
tools (Read/Grep/Glob) never prompt, so they kept working — masking the gap. This also
explains #62's denials (they were `Edit`/`Write` on the workflow file, same cause) — there
is no special "workflow-file guard"; edits were simply disallowed. My first #69 pass got
this backwards; corrected here.

## Fix

- **Add `Edit,Write` to `--allowedTools`.** Traces directly to the #50 denials. Safe: they
  only touch the working tree (sandbox blocks writes outside it); branch protection + human
  review are the real containment.
- **Fail-loud gate** (satisfies #62 + Carl's "make it noisy" ask): after the run, require a
  PR that closes the issue (GraphQL `closedByPullRequestsReferences`); no PR → red job. When
  there's no PR AND the transcript touched `.github/workflows/**`, name the specific reason
  (editable but not pushable — `workflows: write` withheld, §3).
- A workflow-editing task now edits fine but fails at `git push` (GITHUB_TOKEN can't push
  workflow changes without `workflows: write`) → no PR → the guard turns it red. Do **not**
  grant `workflows: write`.

Validation: a throwaway `ready` issue needing normal file edits should now produce a
branch + PR with 0 denials on the edit/ship path. Only exercisable after merge — `issues.
labeled` runs the workflow from the default branch.
