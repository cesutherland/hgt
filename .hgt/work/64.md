## Task (verbatim from the issue body)

## Context
Executor runs only surface stats (`num_turns`, `cost`, `permission_denials_count`) —
the full Claude transcript is written to `/home/runner/work/_temp/claude-execution-
output.json` on the runner, then evaporates when the job ends. When a run misbehaves
(e.g. #61: 14 denials, no PR, green check) there's no way to see *why* after the fact.
The transcript is the stream-json log of every turn, tool_use, tool_result, and denial —
exactly the missing detail. SPEC §6 parks observability in Phase 3; this is its first
concrete need.

## Scope
Add a step to `hgt-execute.yml` that uploads the transcript as a **build artifact**
(not `show_full_output` into the logs):

```yaml
- name: Upload Claude transcript
  if: always()                     # capture on failure too
  uses: actions/upload-artifact@<pin-sha>   # v4, SHA-pinned per §3
  with:
    name: claude-transcript-${{ github.event.issue.number }}
    path: /home/runner/work/_temp/claude-execution-output.json
    retention-days: 7
```

Confirm the temp path is stable (or read it from an action output) before pinning it.

## Why artifact, not `show_full_output`
Artifact = deliberate download, structured/greppable, expires, redactable. Log-dump =
firehose into Actions logs. This posture is correct **regardless of repo visibility** —
it does not rely on public/private as the safety control. §3's real invariant is that
the executor holds nothing worth leaking (powerless-reader / privileged-actor split);
repo visibility is only a blast-radius reducer. Note in the workflow comment that when
the executor eventually touches sensitive context, the transcript needs redaction at
capture — visibility toggles are not the mechanism.

## Non-goals

- Redaction implementation — flagged for when snapshots/secrets land (#39/#40/#41);
